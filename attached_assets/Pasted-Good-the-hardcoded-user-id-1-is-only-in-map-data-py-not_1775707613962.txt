Root cause: The ReportSelectionPanel.tsx was completely disconnected from the backend's checkout-state system. It fetched public pricing, then blindly called generate-free-report for everyone — no email collection for guests, no credit awareness for subscribers, no Stripe redirect for paid purchases.
What the new panel does:
On mount (and whenever the selected report changes), it calls GET /report-pricing/checkout-state?report_type=X — the backend returns one of four states with all the UI config pre-computed:
Guest → Shows email input field, full price, CTA is "Pay $X & generate" → redirects to /studio-report-checkout which creates a Stripe session and auto-creates their account on payment success.
Free tier (logged in, no subscription) → Shows email on file, full price, "Subscribe & save 20%+" secondary CTA → same Stripe checkout path.
Subscriber with credits → Shows credit allocation badge ("3 of 5 remaining"), price shows "$0 (included)", CTA is "Generate report (use 1 credit)" → calls /generate-free-report directly, no payment.
Subscriber, no credits → Shows discounted overage price with strikethrough original, upsell message, "Upgrade plan" secondary CTA → Stripe checkout at the discounted rate.
Backend status: The /checkout-state endpoint was already correctly implemented — it uses the fungible credit model (is_paid_tier + has_credits) and doesn't touch the broken is_report_included_for_tier() function. The /studio-report-checkout endpoint already handles both guest and authenticated users with proper email resolution. No backend changes needed for this flow.
