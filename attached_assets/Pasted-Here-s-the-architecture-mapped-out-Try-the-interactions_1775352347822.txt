Three things to call out from the mockup:
The layer toggle is the key UX decision. Both layers default ON when the trip is live, but the user can strip them. Routes off gives a clean pin-only overview. Expert notes off keeps it minimal for someone who already knows the plan. This also maps cleanly to your existing architecture — activities and transport as toggleable layers on the itinerary card, now extended to the map view.
Transit mode swap is inline, not a settings page. Tapping the connector ("12 min walk · Free") drops down alternatives with time and cost right there. When they pick a different mode, it updates the route line on the map and the connector text simultaneously. The "Open in Maps" link on each connector constructs the deep link with the selected mode — so if they swap to taxi, it opens Maps with driving directions, not walking.
Expert Notes are the Traveloure moat. Without them, this is just Google Maps with pins. With them, every stop has context that no map app can provide — Yuki telling you to use the side entrance, or that the matcha set isn't on the English menu. These notes come from the expert review phase and display both as bubbles on the map and inline under each activity.
For implementation, the data flow stays clean: the map component reads the same activity array and transport legs that the list view uses. No new data structures needed — just a lightweight map renderer (could be a static tile + SVG overlay for v1, or a minimal Google Maps embed) with pins plotted from activity.lat/lng and polylines from the transport leg coordinates, colored by mode.

A few key decisions baked in that are worth highlighting:
Mode swap drives the URL, not the other way around. When the user taps a transport connector and picks "taxi" instead of "walk," the openInMaps call rebuilds with travelmode=driving automatically. It's a function call, not a pre-built href — so the link is always in sync with whatever mode is currently selected.
Waze is conditional. It only appears as an option when the selected mode is driving-compatible (taxi, private car, rickshaw, tuk-tuk). If someone has Waze as their default but the leg is a walking leg, it silently falls back to Google/Apple rather than showing an error.
Full-day route has a limitation worth knowing. Google Maps multi-stop URLs only accept one travelmode for all legs. Since your itinerary mixes modes (walk → taxi → bus), the "Open Full Day in Maps" button uses driving as a route overview. The per-leg "Open in Maps" on each transport connector is where mode-accurate navigation lives. This is actually the right UX — on the ground, people navigate one leg at a time, not all four stops at once.
Market-specific modes (rickshaw, tuk-tuk) map to driving. Maps apps don't know about rickshaws, but the driving route is correct for the road network. The Traveloure connector still shows the actual mode with the right icon and local pricing — the deep link just gets them on the right road.
