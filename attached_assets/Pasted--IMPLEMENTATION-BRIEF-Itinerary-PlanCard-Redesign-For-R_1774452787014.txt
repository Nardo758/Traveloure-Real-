# IMPLEMENTATION BRIEF: Itinerary PlanCard Redesign
## For Replit Agent — Read This Entire Document Before Writing Any Code

**What this is:** A complete specification for redesigning the itinerary card system. The current PlanCard component in `client/src/pages/dashboard.tsx` (lines ~95–291) needs to be replaced with a two-section architecture that separates Activities from Transport, supports collaboration tracking, and exports to Maps/Calendar as layers.

**When this triggers:** This card is generated the moment a user creates an itinerary. Every itinerary gets a PlanCard. The card is the primary interface for viewing, sharing, collaborating on, and navigating a trip.

---

## 1. CORE ARCHITECTURE CHANGE

### Current Problem
The existing PlanCard is a static display card with a photo hero, vibe tags, and action buttons. Transport data was bolted on as a separate "Trip Transport" summary box and a "Ground Transportation" sidebar widget — three disconnected transport sections that don't relate to each other or to the activities timeline.

### New Architecture
The card has **two distinct sections with a deliberate sequence**:

```
PlanCard
├── Hero (photo, title, destination, dates, status, share/export buttons)
├── Stats Row (days, activities count, transit legs, total transit time)
├── Day Selector (tab per day)
├── Section Tabs
│   ├── "Activities" tab (always accessible)
│   ├── "Transport" tab (LOCKED until activities are finalized)
│   └── "Changes" button (opens changelog panel)
├── Section Content
│   ├── IF activities: Timeline list of activities with type badges, status, comments, change indicators
│   └── IF transport: Summary bar + transport legs with mode/duration/cost + accept/reject for suggestions
└── Maps CTA (Open in Google Maps / Apple Maps with layers)
```

**Critical Rule:** The Transport tab shows a lock icon and "finalize activities first" message when ANY activity in the selected day has `status: "pending"`. Transport can only be filled in AFTER all activities for that day are confirmed. This is because transport routes depend on knowing the final activity locations.

## 2. DATA MODEL CHANGES

### Add to the existing trip/itinerary schema:

```typescript
// Each day in an itinerary has activities AND transport segments
interface ItineraryDay {
  dayNum: number;
  date: string;           // "Sat, Jan 31"
  label: string;          // "Arrival & Asakusa" — auto-generated from activities
  activities: Activity[];
  transports: TransportLeg[];
}

interface Activity {
  id: string;
  time: string;           // "14:00"
  name: string;
  location: string;
  type: 'dining' | 'attraction' | 'shopping' | 'transport' | 'activity' | 'accommodation';
  status: 'confirmed' | 'pending' | 'suggested';
  cost: number;
  latitude: number;
  longitude: number;
  comments: Comment[];
  changes: ChangeRecord[];
}

interface TransportLeg {
  id: string;
  fromActivityId: string;  // References Activity.id
  toActivityId: string;    // References Activity.id
  mode: 'walk' | 'train' | 'taxi' | 'car' | 'bus' | 'bike' | 'shuttle' | 'ferry';
  duration: number;        // minutes
  cost: number;
  currency: string;        // "USD", "JPY", etc.
  line?: string;           // "Narita Express → Asakusa Line"
  status: 'confirmed' | 'suggested';
  suggestedBy?: 'ai' | 'expert' | 'user';
}

interface ChangeRecord {
  id: string;
  who: string;            // "Expert Yuki", "Friend Mika", "AI", "You"
  action: string;         // "Moved Senso-ji to Day 1 for golden hour light"
  when: string;           // ISO timestamp
  type: 'edit' | 'suggest' | 'ai' | 'confirm' | 'reorder' | 'add' | 'remove';
  role: 'owner' | 'expert' | 'friend' | 'ai';
}

interface Comment {
  id: string;
  author: string;
  text: string;
  timestamp: string;
  role: 'owner' | 'expert' | 'friend';
}
```

### Database Migration Needed
Add these tables/columns to support the new architecture:

1. `transport_legs` table — stores transport segments between activities
2. `itinerary_changes` table — stores the change log with who/what/when/role
3. `activity_comments` table — stores comments on individual activities
4. Add `latitude`, `longitude` columns to the activities table if not present
5. Add `suggested_by` column to track AI vs expert vs user suggestions

## 4. SHARING & COLLABORATION

### Role-Based Permissions
When a PlanCard is shared, the recipient's capabilities depend on their role:

| Capability | Owner | Expert | Friend |
|-----------|-------|--------|--------|
| View activities | ✅ | ✅ | ✅ |
| View transport | ✅ | ✅ | ✅ |
| Add comments | ✅ | ✅ | ✅ |
| Suggest activities | ✅ | ✅ | ✅ |
| Reorder activities | ✅ | ✅ | ❌ |
| Add/remove activities | ✅ | ✅ | ❌ |
| Edit activity details | ✅ | ✅ | ❌ |
| Fill in transport | ✅ | ✅ | ❌ |
| Confirm/finalize | ✅ | ❌ | ❌ |

### Change Tracking
**Every modification is logged.** The change log stores:
- WHO made the change (name + role)
- WHAT they changed (human-readable description)
- WHEN (timestamp)
- TYPE (edit, suggest, ai, confirm, reorder, add, remove)

Display color coding for change types:
- Blue dot = expert edit
- Purple dot = friend suggestion  
- Green dot = AI auto-suggestion
- Amber dot = owner confirmation

### Transport Filling Sources
Transport legs can be populated by multiple sources (in priority order):
1. **AI auto-suggests** — Based on activity locations, time gaps, and user preferences (walking vs transit preference, budget level). These appear with status "suggested" and suggestedBy "ai"
2. **Expert recommends** — Expert fills in transport as part of their service. Status "suggested", suggestedBy "expert"
3. **User picks manually** — User selects mode for each leg. Status "confirmed"
4. **User accepts suggestion** — User clicks "Accept" on AI/expert suggestion, changing status to "confirmed"

## 5. MAP & CALENDAR EXPORT

### This is the most important feature. The card must export as a control center.

### Map Export (Google Maps / Apple Maps)
When "Open in Maps" is clicked, generate a deep link or file that creates TWO layers:

**Layer 1: Activities (Pins)**
- Each activity becomes a pin/marker on the map
- Pin color matches activity type (dining=amber, attraction=blue, shopping=pink, transport=green)
- Pin label shows activity name + time
- All pins for the selected day are shown

**Layer 2: Transport (Routes)**  
- Each transport leg becomes a route/direction between two pins
- Route style varies by mode:
  - Walk = dashed green line
  - Train = solid blue line
  - Taxi/car = solid amber line
  - Bus = solid purple line
- Route label shows duration + mode

**Implementation approach:**
- For Google Maps: Generate a URL with waypoints for directions, OR create a KML file with two folders (Activities, Transport) that the user can import into Google My Maps
- For Apple Maps: Generate a `.maps` link or use MapKit JS for the in-app preview
- The in-app Map Control Center view should use a real map library (Mapbox GL JS or Google Maps JavaScript API) to show the actual map with toggleable layers

### Calendar Export
When "Add to Calendar" is clicked, generate an .ics file or deep link that creates:
- One calendar event per activity (with location, time, duration)
- One calendar event per transport leg (with "Transit: [mode] from [A] to [B]" as title, duration as the event length)
- Events should include:
  - Location coordinates for map integration
  - Description with any notes/tips
  - Correct timezone for the destination

```
Example .ics structure for one day:

EVENT: 14:00-15:30 "Airport Arrival" @ Narita International (35.7647, 140.3864)
EVENT: 15:30-16:30 "🚆 Train: Narita Express → Asakusa" (transit event)
EVENT: 16:30-17:45 "Senso-ji Temple" @ Asakusa, Taito City (35.7148, 139.7967)  
EVENT: 17:45-17:48 "🚶 Walk to Nakamise" (transit event)
EVENT: 18:00-19:15 "Nakamise Shopping Street" @ Asakusa (35.7118, 139.7963)
EVENT: 19:15-19:23 "🚶 Walk to Ichiran" (transit event)
EVENT: 19:30-20:30 "Ichiran Ramen" @ Asakusa (35.7108, 139.7955)
```

## 6. TEMPLATE-ADAPTIVE BEHAVIOR

The card layout adapts based on the experience type. The core two-section (Activities + Transport) architecture stays the same, but certain UI elements change:

### Travel Template
- Stats row shows: Days | Activities | Transit Legs | Transit Time
- Activity types include: dining, attraction, shopping, transport
- Transport modes include all options
- Standard behavior as described above

### Wedding Template  
- Stats row shows: Days | Events | Vendors Confirmed (8/12) | Guest Groups
- "Activities" tab is labeled "Events & Vendors"
- Activity types include: ceremony, reception, rehearsal, vendor_setup, guest_activity
- Transport section focuses on: Guest shuttles (with capacity), vendor arrivals, bridal party logistics
- Additional indicator: Vendor confirmation status
- Change log emphasizes vendor coordination changes

### Corporate Template
- Stats row shows: Days | Sessions | Attendees | Venues
- "Activities" tab is labeled "Agenda"
- Activity types include: keynote, breakout, networking, team_building, meal
- Transport section focuses on: Airport pickups (staggered), venue transfers (group coach)
- Additional indicator: Attendee group assignments
- Transport legs may have "capacity" field for group logistics

### Proposal Template
- Stats row shows: Moments | Vendors | Backup Plans | Budget
- Minimal transport section (usually single-evening)
- Emphasis on timing precision and weather contingency
- Change log is hidden from the person being proposed to (obviously)

### Date Night Template
- Simplified card — usually single evening
- Stats row shows: Stops | Total Time | Budget
- Transport section is streamlined (fewer legs)
- Emphasis on reservation times and peak timing

## 9. WHAT NOT TO DO

- **Do NOT** put transport data inline between activities. Activities and Transport are separate sections.
- **Do NOT** allow transport editing before activities are confirmed for that day.
- **Do NOT** create separate "Trip Transport" summary widgets disconnected from the card.
- **Do NOT** use a "Ground Transportation" booking sidebar. Transport booking happens within the Transport section of each day.
- **Do NOT** mix the map view with the card view — they are two distinct views toggled by a button.
- **Do NOT** lose the photo hero visual appeal. This is a "balanced overview" — logistics data is layered into the existing attractive design, not replacing it.
- **Do NOT** implement all template types at once. Start with Travel template, then add Wedding and Corporate.
